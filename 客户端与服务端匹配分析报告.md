## 1. 总体评估

### 匹配度评分：约 60%

| 维度 | 匹配度 | 说明 |
|------|--------|------|
| 协议核心操作码 | ~70% | 核心收发匹配，但存在约 20 个服务端发送的包客户端无处理 |
| 自定义功能 | ~0% | 服务端 28 个 Custom 功能全部无客户端对应实现 |
| 加密层 | ~50% | SimpleModulus 理论兼容，但 DES_XEX3 / HackCheck 无客户端对应 |
| 数据结构 | ~90% | 核心包结构匹配，但 Login 包存在 1 字节不匹配 |
| 编译配置 | ~40% | GAMESERVER_UPDATE 头文件与 vcxproj 互相矛盾，功能标志命名完全独立 |
| 数据文件 | ~80% | 大部分 .dat 配置可用，但需要手动同步版本号/序列号 |

### 核心结论

**服务端和客户端不是"完整完美匹配"的。** 两者虽然可以建立连接并运行基础功能，但在多个关键层面存在不匹配。主要问题包括：

- 服务端的 28 个 Custom 自定义功能全部没有客户端实现
- 登录包存在 1 字节结构差异（LanguageCode 字段）
- 加密路径存在分歧（服务端启用 DES_XEX3，客户端未实现）
- HackCheck XOR 加密服务端有但客户端没有
- 版本号/序列号依赖外部配置文件手动同步

## 3. 加密与安全层匹配分析

### 3.1 加密路径选择

服务端 `GAMESERVER_UPDATE` 决定使用哪条加密路径：

| 条件 | 服务端加密 | 客户端加密 | 匹配状态 |
|------|-----------|-----------|---------|
| `>=701` (即 803) | DES_XEX3 (Crypto++ Triple DES ECB) | ASIO challenge-response + **无载荷加密** | **不匹配** |
| `<701` (即 502) | SimpleModulus（经典算法） | SimpleModulus（兼容 .lib） | **理论匹配** |

**关键问题**：服务端 vcxproj 将 `GAMESERVER_UPDATE` 覆盖为 502，所以理论上生产编译使用 SimpleModulus。但服务端 stdafx.h 的默认值是 803，如果有开发者直接使用默认值编译，就会使用 DES_XEX3 路径，而客户端没有实现 DES_XEX3 解密。

### 3.2 SimpleModulus 密钥兼容性

| 项目 | 服务端 | 客户端 |
|------|--------|--------|
| 加密块大小 | 8 字节 | 8 字节 |
| 密钥元素数 | 4 DWORD | 4 DWORD |
| 加密后大小 | 11 字节 | 11 字节 |
| 密钥文件头 | `ENCDEC_HEADER.header = 4370 = 0x1112` | `CHUNKID_ONEKEY = 0x1112` |
| 密钥文件加载 | `*.dat` 文件 | `Data\\Enc1.dat`, `Data\\Dec2.dat` |
| 密钥文件 XOR | `m_SaveLoadXor[4]`（服务端源码可见） | `s_dwSaveLoadXOR[4]`（在闭源 .lib 中） |

**潜在不匹配**：服务端保存/加载密钥时使用的 XOR 混淆码在 `PacketManager.cpp` 中定义，而客户端对应的 XOR 码在闭源的 `SimpleModulus.lib` 中。**这两个值必须完全一致才能正确加载密钥文件。** 由于客户端 .lib 是预编译的闭源库，无法验证这两个值是否匹配。

### 3.3 HackCheck XOR 加密

| 项目 | 服务端 | 客户端 |
|------|--------|--------|
| 实现 | `HackCheck.cpp` (EncDecKey1/EncDecKey2) | **不存在** |
| 加密算法 | `out = (in + EncDecKey2) ^ EncDecKey1` | **无对应实现** |
| 密钥来源 | CustomerName + ServerSerial 派生 | **不适用** |
| MHP 二层加密 | `..\Data\Hack\MHPServer.ini` | **不适用** |

**状态：不匹配**。服务端配置了 HackCheck XOR 加密层（`PROTECT_STATE==0` 时使用免费密钥 `EncDecKey1=0xE2, EncDecKey2=0x76`），但客户端源码中完全没有对应的 HackCheck 解密代码。如果服务端启用了这个加密层，客户端收到的所有包都是乱码。

### 3.4 登录包大小不匹配

这是最具体的二进制不匹配问题：

| 字段 | 服务端协议头 (Language=1, UPDATE>=803) | 客户端 (ProtocolSend.h) |
|------|---------------------------------------|----------------------|
| account[10] | 偏移 0 | 偏移 0 |
| password[20] | 偏移 10 | 偏移 10 |
| TickCount (DWORD) | 偏移 30 | 偏移 30 |
| ClientVersion[5] | 偏移 34 | 偏移 34 |
| ClientSerial[16] | 偏移 39 | 偏移 39 |
| **LanguageCode** | **偏移 55, 1 字节** | **客户端不存在** |
| **总大小** | **56 字节** | **55 字节** |

**影响**：服务端读取的 LanguageCode 字节实际上是客户端 Serial 之后的内存垃圾数据。如果该内存区域恰好为 0，LanguageCode=0 可能碰巧正常工作，但这属于未定义行为。

## 5. 编译配置与功能标志匹配分析

### 5.1 核心编译常量对比

| 常量 | 服务端 stdafx.h | 服务端 vcxproj | 客户端 |
|------|----------------|----------------|--------|
| `GAMESERVER_UPDATE` | **803** (S8) | **502** (S5) | 未定义 |
| `GAMESERVER_LANGUAGE` | **1** | **0** | 未定义 |
| `GAMESERVER_TYPE` | 0 | 0 (GS) / 1 (GSCS) | 未定义 |
| `GAMESERVER_EXTRA` | 0 | 未覆盖 | 未定义 |
| `NEW_PROTOCOL_SYSTEM` | 1 | 未覆盖 | **已定义** (值不同) |
| `_WIN32_WINNT` | `_WIN32_WINNT_WINXP` | — | 未定义 |
| `_USE_32BIT_TIME_T` | 未定义 | — | **已定义** |

### 5.2 严重矛盾：GAMESERVER_UPDATE

**这是项目中最大的配置矛盾：**

- `stdafx.h` 第 29 行：`#define GAMESERVER_UPDATE 803`（Season 8 Episode 3）
- `GameServer.vcxproj` 第 99 行：`GAMESERVER_UPDATE=502`（Season 5 Episode 2）

头文件中的 `#ifndef` 守卫意味着如果项目文件中定义了该值，头文件中的默认值被忽略。**实际编译使用的是 502（Season 5），而非 803（Season 8）。**

然而，`stdafx.h` 中的条件编译广泛使用 `>=701` 分支（例如切换到 DES_XEX3 加密、启用扩展 HP/MP/SD 视图字段等）。当实际使用 502 编译时：

- `#if(GAMESERVER_UPDATE>=701)` → **False**（因为 502 < 701）
- 使用 SimpleModulus 加密路径
- 不启用扩展 HP/MP/SD 视图字段

这意味着**要完美匹配客户端**，服务端必须使用 502 编译（即 Release_EX502 配置），而不能使用 stdafx.h 默认的 803。

### 5.3 功能标志命名差异

服务端和客户端使用**完全不同的命名规范**，没有共享系统：

| 服务端 (stdafx.h) | 客户端 (Defined_Global.h) |
|--------------------|---------------------------|
| `GAMESERVER_*` 前缀 | `PJH_*`, `ASG_ADD_*`, `KJH_*` 等 |
| 数值开关 (1=ON, 2=OFF) | `#ifdef` 风格 |
| 统一在 stdafx.h 管理 | 分散在多个头文件 |
| `NEW_PROTOCOL_SYSTEM 1` (带值) | `NEW_PROTOCOL_SYSTEM` (无值) |

**唯一重叠的标志**：`NEW_PROTOCOL_SYSTEM`。但语法不同（服务端用 `#if(NEW_PROTOCOL_SYSTEM==1)`，客户端用 `#ifdef NEW_PROTOCOL_SYSTEM`），两者不是直接互换的。

## 6. 数据结构兼容性分析

### 6.1 核心包结构匹配表

| 包类型 | 发送方向 | 服务端结构 | 客户端结构 | 匹配状态 |

| 登录 | C→S | PMSG_CONNECT_ACCOUNT_RECV (56字节 + 头) | PMSG_CONNECT_ACCOUNT_SEND (55字节 + 头) | **不匹配** (差 1 字节) |

| 移动 | C→S | x + y + path[8] (10字节, 新协议) | x + y + path[8] (10字节, 新协议) | **匹配** |
| 攻击 | C→S | PBMSG_HEAD + index[2] + action + dir (7字节) | PBMSG_HEADER + KeyH/L + Action + Dir (7字节) | **匹配** |
| 物品(线路格式) | 双向 | ItemByteConvert (12字节/个) | ConvertItemType (12字节/个) | **匹配** |
| 聊天 | 双向 | name[10] + message[60] (70字节) | name[10] + message[60] (70字节) | **匹配** |

| NPC 商店购买 | C→S | 物品索引 + 数量 | 物品索引 + 数量 | **匹配** |

### 6.2 关键不匹配：登录包

```
服务端期望 (56字节):  account[10] + password[20] + TickCount[4] + Version[5] + Serial[16] + LanguageCode[1]
客户端发送 (55字节):  account[10] + password[20] + TickCount[4] + Version[5] + Serial[16]
                                                    ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                                                    最后 1 字节 LanguageCode 缺失
```

服务端在 `GAMESERVER_LANGUAGE==1 && GAMESERVER_UPDATE>=803` 时读取 56 字节载荷，但客户端只发送 55 字节。服务端读取的 LanguageCode 来自栈/堆上的未初始化内存。

### 6.3 对象/角色结构说明

服务端 `OBJECTSTRUCT`（`User.h`）和客户端 `OBJECT`（`w_ObjectInfo.h`）/ `CHARACTER`（`w_CharacterInfo.h`）是**完全不同目的的结构**：

- 服务端：游戏逻辑状态（连接、战斗、背包、任务、公会等，数百个字段）
- 客户端：渲染与显示状态（位置、动画、模型、UI 数据等）

**无需匹配**。两者通过协议包交换数据，不直接共享内存结构。

## 7. 数据文件与资源配置分析

### 7.1 版本号与序列号

| 项目 | 服务端 | 客户端 |
|------|--------|--------|
| 版本号来源 | `GameServerInfo - 连接信息.dat` 中的 `m_ServerVersion` | `Main.info` 文件中的 `ClientVersion` |
| 序列号来源 | `m_ServerSerial` (同配置文件) | `Main.info` 中的 `ClientSerial` |
| 代码中硬编码默认值 | 无 | `"TbYehR2hFUPBKgZj"` (WSclient.cpp:114) |
| 验证方式 | `memcmp` 精确匹配 | 发送原始字节 |

**状态**：依赖外部配置文件手动同步。如果服务端 `GameServerInfo - 连接信息.dat` 中的版本号/序列号与客户端 `Main.info` / `Data\Local\` 中的不一致，登录会被拒绝。

### 7.2 SimpleModulus 密钥文件

| 文件 | 在服务端位置 | 在客户端位置 |
|------|-------------|-------------|
| 加密密钥 (C→S) | `Enc1.dat` (服务端用此解密) | `Data\Enc1.dat` (客户端用此加密) |
| 解密密钥 (S→C) | `Dec2.dat` (服务端用此加密) | `Data\Dec2.dat` (客户端用此解密) |

**必须匹配**：这些文件需要在两端保持一致。由于密钥文件使用 XOR 混淆存储，如果服务端的 `m_SaveLoadXor` 与客户端闭源 .lib 中的 `s_dwSaveLoadXOR` 不一致，即使文件内容相同也会解密失败。

### 7.3 配置文件（GameServerInfo）

`MuServer\GameServer\DATA\GameServerInfo - *.dat` 中的配置项定义了服务端的行为参数，客户端对这些参数无直接感知。需要确保：

- `版本号` 与客户端 `Main.info` 一致
- `序列号` 与客户端 `Main.info` 一致
- 活动开关与客户端支持的 UI 一致

## 9. 修复建议

### 9.1 必须修复（P0）

**1. 修正登录包结构**

在客户端 `ProtocolSend.h` 的 `PMSG_CONNECT_ACCOUNT_SEND` 中添加 LanguageCode 字段：

```cpp
struct PMSG_CONNECT_ACCOUNT_SEND {
    char account[10];
    char password[20];
    DWORD TickCount;
    BYTE ClientVersion[5];
    BYTE ClientSerial[16];
    BYTE LanguageCode;  // 添加此行
};
```

或在服务端条件编译中排除该字段（当客户端不支持时）。

**2. 统一 GAMESERVER_UPDATE**

确认实际编译使用的版本。如果服务端用 Release_EX502（502）编译，确保 `stdafx.h` 中的条件编译逻辑正确反映此版本。如果目标是 803/S8，则需要在客户端实现对应的新功能。

**3. 对齐加密路径**

选择一条加密路径，确保两端一致：
- **选项 A**：使用 SimpleModulus（502 路径），关闭 DES_XEX3 和 HackCheck
- **选项 B**：在客户端实现 DES_XEX3 解密（803 路径）
- **选项 C**：完全禁用 NEW_PROTOCOL_SYSTEM，使用经典 SimpleModulus

### 9.2 建议修复（P1）

**4. 为 28 个 Custom 功能添加客户端 UI**

根据使用需求优先级，逐步实现客户端的 Custom 功能 UI。这些功能是服务端相对于官方 MU 的增值特性，没有客户端支持将完全不可用。

**5. 添加缺失的客户端协议处理器**

对于服务端已发送但客户端不处理的操作码，根据需求添加对应的 `Receive*` 函数：

| 优先级 | 操作码 | 所需工作量 |
|--------|--------|-----------|
| 高 | 0xEC (生命/攻速) | 小（标准 S8 功能） |
| 高 | 0x4E (穆恩系统) | 中（新 UI + 协议） |
| 高 | 0xED (公会匹配) | 中（新 UI + 协议） |
| 中 | 0x4C (采矿系统) | 小（迷你游戏 UI） |
| 中 | 0x4D (MuRummy) | 中（卡牌游戏 UI） |
| 低 | 0xAE, 0xBF 等 | 依功能而定 |

### 9.3 长期建议（P2-P3）

**6. 建立统一的功能标志系统**

创建共享的功能标志头文件，或者建立文档记录两端功能标志的对应关系。

**7. 密钥文件验证**

编写工具验证 `Enc1.dat` / `Dec2.dat` 在两端的加载结果是否一致。

**8. 建立协议变更流程**

任何协议变更（新增操作码、修改结构）都需要在两端同步进行。
